# Contributing

[中文](./CONTRIBUTING.md)

Thanks for helping improve Olyq.

## Local Development

Install dependencies:

```bash
pnpm install
```

Start the Chromium extension development build:

```bash
pnpm dev:extension:chromium
```

Start the Firefox extension development build:

```bash
pnpm dev:extension:firefox
```

## Validation

Before opening a pull request, run the relevant checks:

```bash
pnpm lint:extension
pnpm typecheck:extension
pnpm test:extension
pnpm build:extension:chromium
pnpm build:extension:firefox
pnpm build:www
```

For the full local check:

```bash
pnpm verify
```

`pnpm verify` runs `pnpm verify:github-workflows` first. That check rejects GitHub Actions references that are not pinned to a full commit SHA or do not keep a version comment.

## Extension Packages

After building both browser targets, create local zip packages with:

```bash
pnpm package:extension
```

The zip files are written to `apps/extension/artifacts/` and named by intent:

- `olyq-chrome-web-store-<label>.zip`: submit this to the Chrome Web Store, or use it for local loading in Chromium-based browsers.
- `olyq-firefox-amo-addon-<label>.zip`: submit this as the Firefox Add-ons / AMO addon package, or use it for Firefox temporary loading.
- `olyq-firefox-amo-source-<label>.zip`: submit this as the Firefox Add-ons / AMO source code package; it is not an install package.

If you only need one target:

```bash
pnpm package:extension:chrome-web-store
pnpm package:extension:firefox-amo-addon
pnpm package:extension:firefox-amo-source
```

Build the release documentation package from the repository root with:

```bash
pnpm package:release-docs
```

## Release Process

Olyq releases are driven by GitHub Actions. Daily development lands in `dev`, and `main` only carries reviewed release state.

1. Run **Prepare Release** in GitHub Actions and enter an `X.Y.Z` or `X.Y.Z-prerelease` version.
2. The workflow updates the root `package.json` and `apps/extension/package.json` from `dev`, runs `pnpm verify` and mock E2E, rebuilds production extension output, and uploads preview artifacts retained for 14 days:
   - `olyq-chrome-web-store-vX.Y.Z-pre.<run_number>.zip`
   - `olyq-firefox-amo-addon-vX.Y.Z-pre.<run_number>.zip`
   - `olyq-release-docs-vX.Y.Z-pre.<run_number>.zip`
3. Download and validate preview artifacts with `docs/release/manual-qa.md`; preview packages are for manual QA only and must not be uploaded to stores or attached to GitHub Releases.
4. After reviewing and merging `release/vX.Y.Z` into `main`, run **Release** from `main` with the same version.
5. The Release workflow rebuilds cleanly from `main`, creates the `vX.Y.Z` tag, packages Chrome, Firefox addon, Firefox source, and release docs zip assets, writes the complete `SHA256SUMS.txt`, creates GitHub artifact attestations, and publishes the GitHub Release page and assets.

Do not push release tags from a local machine. The GitHub Release page, release notes, zip assets, checksum, and provenance are all generated by the **Release** workflow.

## Store Submission Packages

Use the package name, not browser guesswork:

- Chrome Web Store: upload `apps/extension/artifacts/olyq-chrome-web-store-${tag}.zip`.
- Firefox Add-ons / AMO: upload `apps/extension/artifacts/olyq-firefox-amo-addon-${tag}.zip` as the addon package, and upload `apps/extension/artifacts/olyq-firefox-amo-source-${tag}.zip` as the source code package.
- Release docs: download `apps/extension/artifacts/olyq-release-docs-${tag}.zip` and follow `chrome-web-store.md`, `firefox-amo.md`, and `manual-qa.md`.

The Firefox AMO source package is only for reviewer rebuilds, not for user installation. Firefox users should receive the `.xpi` produced by Mozilla's review / signing flow. Do not add Chrome Web Store or Firefox AMO credentials to the GitHub Release workflow; store publishing stays behind separate accounts, permissions, and secrets.

## Product Assets

Product images use `assets/product/` as the source of truth. `apps/www/public/product/` is only a static copy synced before website builds; do not edit or commit that directory directly.

README images, website home / features grids, blog covers, and OG / Twitter sharing images must use real extension UI screenshots or Olyq website semantic images. The four product-grid screenshots are generated by `apps/extension/scripts/generate-website-product-screenshots.mjs` with a `1280×800` viewport and `deviceScaleFactor: 2`, producing `2560×1600` Retina PNG files; do not add new outward-facing `1280×800 @1x` product screenshots.

`olyq-store-*` store posters and `olyq-promo-*` promotional tiles are only for Chrome Web Store / Firefox Add-ons store materials. Do not reference them as README, website, blog, or sharing-meta product screenshots.

Public copy should describe Olyq as an open-source multi-model AI workspace for the browser, using only shipped or documented capabilities. Do not present it as a store-listed product, hosted model service, knowledge-base RAG product, desktop app, team workspace, or Cherry Studio browser edition / fork.

Common commands:

```bash
pnpm sync:product-assets
cd apps/extension && pnpm generate:store-posters
cd apps/extension && pnpm generate:store-promos
cd apps/extension && pnpm generate:website-product-screenshots
TINIFY_API_KEY=... pnpm optimize:product-assets
```

The generation scripts write to `assets/product/` and sync the website copy. Website `dev` / `build` also sync once before Vite starts, so local previews do not miss images.

`optimize:product-assets` calls the TinyPNG / Tinify API for `assets/product/*.png`, then refreshes the website copy after successful compression. Pass the API key only through `TINIFY_API_KEY`; do not commit it to the repository, scripts, or CI logs. Preview the target files first:

```bash
pnpm optimize:product-assets -- --dry-run
pnpm optimize:product-assets -- --include=olyq-page-context --limit=2
```

The optimization script is not part of the default `verify` flow, so local validation and CI do not accidentally consume TinyPNG quota.

## Pull Request Expectations

Keep changes focused. A pull request should usually cover one feature, fix, or documentation update.

Daily development and integration land in `dev`; use **Prepare Release** to create a release PR targeting `main`.

Please explain:

- What changed.
- Which browser or runtime area is affected.
- Whether permissions, storage, privacy, provider calls, MCP, backup / sync, or release artifacts are affected.
- Which validation commands passed.

## Browser Extension Boundaries

Do not change these boundaries casually:

- The toolbar action opens the side panel / sidebar directly; do not reintroduce a user-visible popup launcher.
- The side panel / sidebar stays the main workspace.
- The service worker stays the routing and background task hub.
- Content scripts stay page-facing.
- Offscreen stays for capabilities that do not belong in the service worker.

Do not expand install-time permissions, host permissions, or `web_accessible_resources` without a clear user benefit and validation.

Olyq is in active development and intentionally avoids hidden fallback behavior, legacy compatibility branches, and extra runtime probing unless the project explicitly chooses that behavior.

## Security And Privacy

Do not include API keys, OAuth tokens, private page content, hidden screenshots, backup archives, or user data in public issues or pull requests.

Use [SECURITY.md](./SECURITY.md) for vulnerability reports and [PRIVACY.md](./PRIVACY.md) for the current data-flow summary.
